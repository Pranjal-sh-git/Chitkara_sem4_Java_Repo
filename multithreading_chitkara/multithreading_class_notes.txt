Introduction to Multithreading (Java)
========================================


--Before learning multithreading, we must understand multitasking.

--Consider the following program:

class Test {

    void fun1() { } //1000
    void fun2() { } // 10000
    void fun3() { } // 10

    public static void main(String[] args) {
        Test t = new Test();
        t.fun1();
        t.fun2();
        t.fun3();
    }
}


Execution Flow (Normal Execution)

First fun1() executes completely, then control comes back to the main method and then fun2() executes completely then again control comes to the main and then fun3() executes completely.

--This is called Sequential Execution.


CPU Idle Problem:
-----------------

--Suppose fun1() contains I/O operation (file reading, data transfer, network call, etc.)

--I/O operation is handled by DMA (Direct Memory Access)
--CPU does not perform data transfer(CPU assign this job to the DMA)
--During I/O operation, CPU remains idle

--This is a waste of CPU time

--This made software developers to think that,efficiency of the processor(CPU) would be increased if processor is made to do some other useful work during this idle state.

--The useful work is nothing but,executing other functions/methods present in the program.


--this need lead to the concept of multitasking..


--Remember that processor is also an electric circuit and at any instance of time,it can execute only one instruction.it can not execute multiple instructions simultaneously unless it is a multi-core processor.


What is Multitasking?
---------------------

--Multitasking is a technique in which CPU executes multiple tasks by switching between them very fast, giving the illusion that tasks are running simultaneously.

Important Point:

--A single CPU core executes one instruction at a time
--But switches between tasks very fast
--This is called Context Switching


--If we apply multitasking then a part of the one fun get executed then control switchs to the another fun now here also some part of second fun get executed then control switchs to the third fun then again control comes back to the fun1 now it continues fun1 from where it had stopped earliar.

--now when we see the output we feel that three fun executing simultaneously.

--Thus,The concept of multitasking come into existence to avoid the idle state of the CPU.



Context Switching:
------------------

--Context Switching is the process of saving the state of one task and switching the CPU to another task.

Example:

--CPU runs fun1 for some time
--Switches to fun2
--Switches to fun3
--Comes back to fun1

--So all functions appear to run simultaneously.


--In multitasking ,a part of a functionality is executed one at a time and in that part how many statement will be executed will be decided by the scheduling.



Scheduling:
----------

--It is the process in which a specific time period(Time Slice) is allocated to a functionality where the control remains in that particular functionality for that specific time period(Time Slice).

--once time period lapsed control switches another functionality with another time slice and so on.

--Scheduling is supervised by the Scheduler(either OS-schedular  or Thread-scheduler).

Types of Scheduler:
-------------------

1. OS scheduler: Works on Processes (different applications)

2. Thread scheduler: Works on Threads (Single application)


---Thread-Schedular in Java is the part of the JVM, that decides which thread should run.

--In Java, thread scheduling is done by JVM Thread Scheduler, but actual execution is performed by OS.

--The time slice is very small (usually milliseconds or nanoseconds depending on OS and scheduler).. thus by the time our eye recognize the execution of one part control switches to another part of the program.




Advantage of multitasking:

--It is invented to avoid Idle states of the cpu.

1. Make the functionalities get executed independently(mostly used because for a small project idle state of a cpu is not a big concern).

2. Better CPU utilization
3. Faster execution
4. Improved performance
5. Reduced response time
6. Allows simultaneous tasks

Multitasking is useful for:

--Games
--Animations
--Video streaming
--Web servers
--Banking systems
--GUI applications



Multiprogramming vs Multitaksing:
---------------------------------

--Multiprogramming and multitasking are similar because both involve multiple programs sharing the CPU. However, in multiprogramming CPU switches when a program waits for I/O operations, while in multitasking CPU switches after a fixed time slice for better response time.

--In other word, Multitasking is an advanced version of multiprogramming where CPU switches between tasks after a fixed time interval instead of waiting for I/O operations.


| Term             | Trigger to Switch |
| ---------------- | ----------------- |
| Multiprogramming | I/O Wait          |
| Multitasking     | Time Slice        |





Types of Multitasking:
----------------------

--Multitasking is of two types:

1. Process based multitasking
2. Thread based multitasking



1. process based multitasking:
------------------------------

--A Process is a program under execution.

Example:

Running Chrome: Process
Running VLC player: Process
Running MS-Word : Process


--The concept of executing more than one program simultaneously which are present in the different location of RAM is known as process based multitasking.

--Here processer has to maintain the address of both the program since control has to switch from one part of ram to another.

--it increase overhead on processor.

--Here OS (OS-scheduler) perform the scheduling.

Example

--Chrome
--VLC Player
--MS Word

--All running at same time.



2. Thread Based multitasking:
-----------------------------

--The concept of executing more than one functionality simultaneously belonging to the same memory domain is known as Thread based multitasking.

--Running multiple tasks within the same program.

--Here Java Thread-scheduler performs the scheduling.

Note: A Thread is the smallest unit of execution(lightweight sub-process) inside a process.


Example:

--Typing in MS Word while auto-save running
--Web server handling multiple users
--A program running with animation.

What is a Thread in Java?

Multithreaded Application:

Concurrency vs Parallelism:
===========================


Concurrency = Managing multiple tasks at the same time
Parallelism = Executing multiple tasks at the same time


Concurrency
-----------

--Concurrency is a situation where multiple tasks are in progress at the same time, but CPU executes them one by one by switching between them very fast.

--Uses context switching
--Usually happens in single-core CPU
--Tasks appear to run simultaneously, but actually they are not

Example

You are:

--Writing an assignment
--Talking on phone
--Drinking tea

--You are not doing all at the exact same time, but switching between tasks, This is Concurrency.


Parallelism:
------------

--Parallelism is a situation where multiple tasks are executed at the exact same time using multiple CPU cores.

--Requires multi-core processor
--Tasks actually run simultaneously

Example

In a restaurant kitchen:

One person making roti
One person making sabji
One person making tea

--All tasks happening at the same time, This is Parallelism.


Concurrency vs Parallelism:
---------------------------


| Concurrency               | Parallelism                 |
| ------------------------- | --------------------------- |
| Tasks appear simultaneous | Tasks actually simultaneous |
| Single CPU core           | Multiple CPU cores          |
| Context switching         | True simultaneous execution |
| Slower than parallelism   | Faster                      |
| Example: Multithreading   | Example: Multiprocessing    |




Real Life Example:
 
--One teacher teaching 3 students one by one: Concurrency

--Three teachers teaching 3 students: Parallelism



Multiprocessing
---------------

--Multiprocessing is a technique where multiple processes execute simultaneously using multiple CPU cores.

--In a multi-core CPU, each core can execute a separate process (application).
--Inside each process, multiple threads can run concurrently.
--These threads share the same memory of the process.


Note: In a multi-core CPU system, the operating system schedules processes and threads on different CPU cores. Each process can contain multiple threads, and these threads can run concurrently or in parallel depending on the number of CPU cores available.


Note: Multithreading provides concurrency and can provide parallelism on multi-core CPU.

Implementing Thread in Java:
============================

--Thread implementation in Java is a two-step process:


1. First of all we have to define a functionality which can be executed as a thread along with the main thread(Define job for a worker)

2. This functionality should be start as a thread.(assign job to worker) using the start() method of java.lang.Thread class.


Runnable interface:
------------------

--The signature of a function using which we implement a thread(or what job a thread has to do) is defined in an interface by name Runnable

--This interface belongs to java.lang package.

This interface has only one abstract method i.e

	public void run();

--For which we have to provide a body (the task a new Thread should perform).

--After providing body we need to execute this funtionality as a new thread(i.e simulteniously with the other part of the program).

--There is a class by name Thread present in java.lang package, which has a method called start(), this start() method is used to execute a given functionality as a seperate thread.

--This start method recognize the run() method of the Runnable interface and then run() method is executed as a seperate individual thread.

--Here thread class is like a worker who has to start the job individually defined by run() method of Runnable interface.

--With the help of run() method we define a job that has to execute as thread, and with the help of Thread class start() method we need to start the job as a seperate individual thread.


Note: Thread class and Runnable interface are the two structures using which we implement Thread based multitasking in Java.


Runnable interface in Java:
----------------------------

interface Runnable{
	public void run();
}

Q/- How to define and implement multiple threads when our class is extending another class other than Thread class.


--If we extends Thread class to create a new Thread, we loose the chance of oo biggest advantage of inheritence i.e we cannot extends another class simmultenoisly,

--To solve the above problem, we use the Runnable interface, so we take a seperate class by implementing the Runnable interface and overrides the run() method and inside the run() method define the functionalty which we want to execute as a seperate thread,

--Then supply the object of that class to the Thread class constructor and create Thread class object and call start() method to start that thread. 

--If we extends Thread class to create a new Thread, We loose the chance of oo biggest advantage of inheritence i.e we cannot extends another class simmultenoisly, we can think it as a design perspective also ,we have to perform our job either by ourself by making ourself as a worker by extending Thread class (IS-A relationship) or only assign a job to the worker (think a thread class as a worker who perform our job defined in Runnable interface run() method) By implementing the Runnable interface and override run() method and hands over that method to the Thread class constructor(to the worker.)



Example:

class MyRunnable implements Runnable {

    public void run() {
        for(int i = 20; i < 40; i++) {
            System.out.println("Inside run() of MyRunnable: " + i);
        }
    }

    public static void main(String[] args) {

        MyRunnable m1 = new MyRunnable();   // Step 1: Create Runnable object

        Thread t1 = new Thread(m1);    // Step 2: Pass Runnable object to Thread

        t1.start();                    // Step 3: Start thread

	
	// Main thread functionality
        for(int i = 20; i < 40; i++) {
            System.out.println("Inside main(): " + i);
        }
    }
}

--Implement Runnable is better than extending Thread because Java supports multiple inheritance through interfaces only.

class SUMThread extends Thread {

    private long startValue;
    private long endValue;
    private long result = 0;

    public SUMThread(long startValue, long endValue) {
        this.startValue = startValue;
        this.endValue = endValue;
    }

    public void run() {
        for (long i = startValue; i <= endValue; i++) {
            result = result + i;
        }

Life Cycle of a Thread (Thread States in Java):
===============================================

--In Java, a thread always exists in one of the following states:


1. New State: Thread object created but start() not called.
2. Runnable State: Thread is ready to run and waiting for CPU. (when start() method is called)
3. Running State: Thread is executing run() method (Got the CPU time).
4. Blocked / Waiting State: Thread is waiting (sleep, wait, I/O etc.)
5. Dead State (Terminated): Thread execution finished.



Note: Once a thread becomes runnable, it can move between Runnable <-> Running <-> Blocked many times.


official Java states (from Thread.State enum) are:

1. Thread.State.NEW
2. Thread.State.RUNNABLE (combines ready + running; JVM doesn't expose a separate "Running" state)
3. Thread.State.BLOCKED (waiting for monitor lock, e.g., synchronized block)
4. Thread.State.WAITING (indefinite wait, e.g., wait(), join() without timeout)
5. Thread.State.TIMED_WAITING (timed wait, e.g., sleep(), join(long), wait(long))
6. Thread.State.TERMINATED

Note: The JVM exposes RUNNABLE (which includes actually executing), BLOCKED, WAITING, TIMED_WAITING, TERMINATED states only as enums. 'Running' is a conceptual sub-state.

isAlive() Method of Thread class:
=================================

--This method checks whether a thread is alive (running or not finished).

public final boolean isAlive();


true: Thread is still running (not finished)
false: Thread has not started yet OR thread has finished execution


Thread Life and isAlive():
-------------------------


| Thread State    | isAlive() |
| --------------- | --------- |
| New             | false     |
| Runnable        | true      |
| Running         | true      |
| Blocked/Waiting | true      |
| Dead            | false     |


Note: The isAlive() method returns true from the time start() is called until the run() method finishes.






Example:
--------

MyThread.java:
--------------

public class MyThread extends Thread {

    @Override
    public void run() {
        System.out.println("Thread is running...");
    }

    public static void main(String[] args) {

        MyThread t1 = new MyThread();

        System.out.println("Before start(): " + t1.isAlive());

        t1.start();

	//Thread.sleep(1000);

        System.out.println("After start(): " + t1.isAlive());
    }
}



Situation:

--Suppose you are downloading a file, and you want to check whether download is still in progress or completed.

Example:


MyThread.java:
-------------


package com.chitkara;

public class MyThread extends Thread{
	
	
	@Override
	public void run() {
	
		System.out.println("Download started..");
		
		try {
			
			Thread.sleep(3000); //simulate download time
			
		} catch (InterruptedException e) {
			e.printStackTrace();
		}
		
		
		System.out.println("Download completed..");
		
	}
	
	
	public static void main(String[] args) throws InterruptedException {
		
		MyThread t1 = new MyThread();
		
		t1.start();
		
		while(t1.isAlive()) {
			
			System.out.println("Downloading...please wait");
			
			Thread.sleep(1000);
		}
		
		
		System.out.println("Now you can open the file..");
		
		
	}
		
}







Thread Priorities:
==================

--Every thread in Java has some priority. It may be the default priority generated by the JVM or explicitly provided by the programmer.

--The valid range of thread priority is 1 to 10. but not 0 to 10.

--1 is the least and 10 is the highest.


Note: Priority should be assigned to the corresponding thread before calling the start() method. We can set priority after start() also, but it may not affect execution because the thread may already be running or finished. So it is recommended to set priority before start().

--Thread class defines the following constant to represent standard priorities:

1. Thread.MAX_PRIORITY---->10
2. Thread.MIN_PRIORITY---->1 
3. Thread.NORM_PRIORITY--->5 //Default

--The Thread-scheduler will use these priorities while allocating the processer to our thread.

--The thread which is having highest priority will get a chance more. whereas the thread which has lower priority will get chance less.

--If two threads have the same priority then we can't expect the exact execution order. it depends on the thread scheduler.

--Thread class defines the following setter and getter methods to get and set the priority of a thread.

1. public final int getPriority();
2. public final void setPriority(int priority);// it allows values from 1 to 10 otherwise we will get an IllegalArgumentException exception.


Note: the default priority for the main thread is 5 and for all remaining threads which started from the main, it will be inherited from parent to child.

--A high-priority thread does not run faster than a lower-priority thread.

--Priority decides importance, not execution order.

Note: Thread priority is not a rule for the thread scheduler, it is just a  hint. so no guarantee for the execution (it may work only if the thread is in the waiting state or if there is a limited CPU time).

--If we are using thread priority for thread scheduling then we should always keep in mind that the underlying platform should provide support for scheduling based on thread priority.


Question: Does MAX_PRIORITY thread always execute first?
Answer: No.

Why?
--Because Java uses Preemptive Priority Scheduling, but it depends on:

OS
JVM
CPU availability

--So priority increases probability, not guarantee.

Thread Starvation
------------------

--Thread Starvation is a situation where a low priority thread does not get CPU time for a long time because high priority threads are continuously running.


Simple Analogy:
---------------

--Imagine a class where a teacher is asking questions.


Topper student	-> High priority
Average student	-> Medium priority

--If teacher always asks questions only to topper, then:

Topper keeps answering
Average and weak student never get chance

This is called Starvation.

Case 2: Fair Scheduling (No Starvation)

--Teacher gives chance to everyone:

First topper
Then average student
Then weak student

--This is called Fair Scheduling or Time Slicing.

--Modern OS uses Fair Scheduling or Time Slicing to prevent starvation.

--Priority decides how often a thread gets CPU. 

| Thread | Priority | CPU Chances     |
| ------ | -------- | --------------- |
| T1     | 10       | Runs more times |
| T2     | 5        | Runs normal     |
| T3     | 1        | Runs less times |

Thread safety in Java:
=====================


--Thread safety means a program behaves correctly when multiple threads access shared data(mutable data) at the same time.

--If multiple threads access shared mutable data without synchronization, it may lead to:

Concurrency issue
Race condition
Data inconsistency

--It is the concept of avoiding multiple threads acting upon the same funtionality simultaneously.

--If multiple threads are trying to operate simultaneously on the same functionality then there may be a chance of data inconsistency problem.

--Concurrency issue lead to the race condition. and race conditions lead to the data inconsistency.

Thread Safety = Avoid data inconsistency when multiple threads access shared resource.


So, Thread safety means designing your code so that it produces correct results even when multiple threads access shared mutable data at the same time.
Without proper synchronization, concurrent modification of shared resources can lead to race conditions and data inconsistency.



Data Inconsistency:
-------------------

--Data inconsistency happens when multiple threads modify the same data at the same time and the final result becomes incorrect.

Example 1: Ticket Booking

Available tickets = 5

Person A wants 4 tickets
Person B wants 3 tickets

If both person book at same time by checking "are there enough tickets?" then : total booked = 7 // WRONG


Example 2: Bank Account

Balance = 5000

ATM withdraw = 4000
UPI payment = 3000

--If both operations happen at the same time without coordination, the final balance can become wrong (e.g., negative or incorrect).
--These issues arise because the operations (check and then act) are not atomic.

This problem is called:

1. Concurrency Issue: Multiple threads working on shared data together.

2. Race Condition: Threads racing to modify data(The result depends on the relative timing of threads).

3. Data Inconsistency: Wrong final result(due to race conditions)


Note: Even a simple i++ is not atomic: it involves read -> modify -> then write steps that can interleave.

--To solve the data inconsistency problem in Java "synchronized" keyword is used.
--So, threadsafety is achieved and race condition is avoided by the help of "synchronized" keyword.

Lock Concept in Java (Synchronization):
========================================

--Synchronization in Java is internally implemented using the Lock concept.

--Every object in Java has a unique lock.
--We can imagine that every object has one key.
--A thread must get that key (lock) to enter a synchronized method/block.
--Only one thread can hold the lock of an object at a time.

Note: Locks are per object, not per method.


Simple Analogy (Real-Life Example):
-----------------------------------

--Imagine a bathroom in a house:

Bathroom = Object
Lock = Key
Person = Thread

--If one person locks the bathroom from inside, other people must wait until the lock is released.

--Same happens in synchronization.


--When a thread enters a synchronized method: it acquires the lock of that object.

--The thread won't release that lock until it completes the synchronized methods, so while that thread is holding the lock for that object. once a synchronized method execution completed then that thread releases that lock automatically.

--When the synchronized method finishes:  it releases the lock automatically
--Lock handling is done by JVM, not by programmer.


Important Rule:

--If one thread is executing a synchronized method on a particular object:

--Other threads cannot execute any synchronized method of that object
--But other threads can execute non-synchronized methods of that object.

Example:

class A {
    synchronized void funA() {}
    synchronized void funB() {}
    void funC() {}
}


--If one thread is executing funA():

--Other threads cannot execute funA() or funB()
--But they can execute funC() simultaniously.

Object Level Lock vs Class Level Lock:
--------------------------------------

--Java has two types of locks:

1. Object Level Lock: Each object has its own lock (It is unique for each object.)

2. Class Level Lock: Each class has one lock (It is unique for each class).


--If a thread executes a static synchronized method, it needs class-level lock.

--Object level lock and class level lock both are independent and there is no link between them.




Note: When a thread is executing a static synchronized method of a class:

--Other threads cannot execute any other static synchronized method of that class at the same time (even if using multiple objects of that class).

But other threads can execute simultaniously:

Synchronized Block:
===================

---If a very few line of code requires synchronization, then it is not recommended to declare the entire method as synchronized. Instead we should enclose those few line of the code inside a synchronized block.


Analogy:

Imagine from Chandigarh to Amritsar there is a highway of 20 km and in that highway someware in the middle there is a narrow bridge for 500 meter where only one vachile can travel then making entire highway to travel only one vehicle.

--So we should lock only the bridge, not the entire highway.

similarly,

--Inside a method, assume 10000 lines code is there, and in the middle somewhere few lines need some Database operation like(update/delete/add) then declaring entire method as synchronized is a worst kind of programming. here it degrades the performance.

--The main advantage of synchronized block over synchronized method is, it reduces the waiting time of the threads and improves the performance of the system.

Note: If we are using synchronized block, we should use it inside any method only, not as a class instance member, otherwise we will get a compile time error.

Difference between synchronized method and synchronized block:
-------------------------------------------------------------

1. Synchronized method locks the entire method, which may reduce performance because threads have to wait even if only a small part of the method needs synchronization.

Synchronized block locks only the required section of code, which improves performance.


2. For non-static synchronized method a thread acquires lock on "this" (the instance on which the method is called).
For static synchronized method the thread acquires the class-level lock to execute the critical section.

In case of a synchronized block, the thread can acquire the lock of:

Current object: synchronized(this) { }  
Any specific object: synchronized(obj) { }  
Current class: synchronized(Common.class) { }
Any other class: synchronized(String.class) { }

--Therefore, synchronized block provides more flexibility in choosing the lock.

public class DeliveryAgent extends Thread {

    private String agentName;
    private Restaurant restaurant;   // Different object we'll lock on

    public DeliveryAgent(String agentName, Restaurant restaurant) {
        this.agentName = agentName;
        this.restaurant = restaurant;
    }

    @Override
    public void run() {

        System.out.println(agentName + " arrived at " + restaurant.getName());

        // Only the critical section (picking up food) needs synchronization
        // We are locking on the Restaurant object.
        synchronized (restaurant) {   
            System.out.println(agentName + " entered kitchen of " 
                               + restaurant.getName() + " (Lock acquired)");

            try {
                Thread.sleep(1500); // Simulating time to pick up and pack food
            } catch (InterruptedException e) {
		e.printStackTrace();
            }

            restaurant.incrementOrdersReady();

            System.out.println(agentName + " picked up order from " 
                               + restaurant.getName() 
                               + ". Total orders picked: " + restaurant.getOrdersReady());

            System.out.println(agentName + " left kitchen of " 
                               + restaurant.getName() + " (Lock released)");
        }

        // Non-critical work (can be done concurrently)
        System.out.println(agentName + " is now delivering food on bike...\n");
    }
}

InterThread communication:
==========================

--It means two synchronized threads communicate each other.

--Inter-Thread Communication is a mechanism by which synchronized threads can communicate with each other to coordinate their actions.


--It allows one thread to suspend its execution and wait for another thread to perform some task (like producing data), and then get notified when the task is done. This helps avoid busy-waiting and improves efficiency.

--Java provides three key methods from the Object class for inter-thread communication:

wait()
notify()
notifyAll()





public final void wait()throws InterruptedException

public final native void wait(long ms)throws InterruptedException

public final void wait(long ms,int ns)throws InterruptedException

public final native void notify()

public final native void notifyAll()








--By using above methods we can gain partial control on the scheduling mechanism which is supervised by the thread-scheduler.

---To gain this partial control the threads should have a sign of mutual understanding between them. they should be able to communicate with each other.

--Whenever we need to suspend a synchronized thread unconditionally then we use wait() method.

--Whenever we need to resume a suspended(waiting) thread then we use notify() method.


--In the inter-thread communication the thread which require updation it has to call wait() method.
--And the thread which performing updation it will call notify() method so that waiting thread will get notification and it continues its execution with those updation.


Note: To call wait() or notify() method on any object, we must have that particular object lock otherwise we will get a runtime exception(IllegalMonitorStateException).


Important Rules for wait(), notify(), and notifyAll():
-----------------------------------------------------

1. These methods belong to the Object class, so any Java object can be used as a communication medium.

2. The calling thread must own the lock of the object on which these methods are called.

3. wait(): Releases the lock of the current object immediately and puts the thread into the waiting state (WAITING or TIMED_WAITING).

4. notify(): Wakes up one thread that is waiting on the same object. It does not release the lock immediately (lock is released only after exiting the synchronized block/method).

5. notifyAll(): Wakes up all threads waiting on the same object.


Q/- Why do we need Inter-Thread Communication?

--Without proper communication, one thread may keep waiting unnecessarily (using sleep() or join()), which is inefficient.

public class Calculator extends Thread {

    private int total = 0;

    @Override
    public void run() {
        
	synchronized (this) {  // Acquire lock on Calculator object

            System.out.println("Child thread started calculation...");

                total += i;

            this.notify(); // Notify waiting thread
        }
    }
}

Difference Between wait() and join():
-------------------------------------


1. 

join(): Wait for a thread to complete its execution
wait(): Wait until notified by another thread.

2. 

join(): Called on a Thread object.
wait(): Called on any shared Object

3. 

Join(): No lock required.
wait(): Must be called from synchronized block/method

4. 

join(): Thread must finish completely
wait(): Can be notified even if work is not fully completed.

5. 

join(): Sequential execution coordination
wait(): Producer-Consumer, resource availability


Key Point:

t.join(): Current thread waits till thread t dies.
obj.wait(): Current thread waits till some other thread calls obj.notify() or obj.notifyAll().



Important Behavior of wait():
-----------------------------

1. When a thread calls wait(), it immediately releases the lock on that object.

2. This allows other threads to enter synchronized blocks/methods on the same object.

3. After being notified, the thread must re-acquire the lock before continuing execution.

Note: A thread can hold multiple locks. wait() only releases the lock of the object on which it is called.



Producer-Consumer Problem in Java:
==================================


--The Producer-Consumer Problem is a classic example of inter-thread communication.

Producer Thread: Produces (generates) data and puts it into a shared resource.
Consumer Thread: Consumes (uses) the data from the shared resource.

The key challenge is to ensure that:

--The Producer does not produce data when the buffer is full (or when data is not yet consumed).
--The Consumer does not consume data when the buffer is empty (or when no new data is available).
--Data is not produced or consumed multiple times.


This coordination is achieved using wait() and notify() methods.


Implementation: without using wait() and notify()
--------------------------------------------------

	public void produce(int i) {

		value = i;
		System.out.println("Producer Produced " + i);

	}

	public int consume() {
		
		return value;
	}
}

			c.produce(i);
			i = i + 1;

			}

		}

	}

}

	Common c;

	public Consumer(Common c) {

		this.c = c;
	}

	@Override
	public void run() {

		while (true) {

			int i = c.consume();

			System.out.println("Consumer Consume :" + i);

			try {
				Thread.sleep(1000);
			} catch (InterruptedException e) {

				e.printStackTrace();
			}

		}

	}

}

Problems in this version:

--Consumer may consume the same value multiple times.
--Producer may overwrite the value before Consumer reads it.
--Consumer may try to consume before any value is produced:  gets old/default value.
--No coordination between threads: Race Condition and Data Inconsistency.


To achieve the proper coordination between Producer and consumer thread, we need to make use of wait() and notify() method.

--First of all producer thread should produce a number after this it has to notify the consumer thread and it(producer thread) go into the wait state.

--Meanwhile, consumer thread should print the produced value on the console and go into the wait state. before going to the wait state it should notify the producer thread so that producer thread will generate a new number and this process continiues.

--To fulfil the above condition we make use of boolean variable flag.

If flag==true--->producer thread should produce the value and consumer thread should wait.

If flag==false---->consumer thread should consume the value and producer thread should wait.


Example:

Common.java:
------------


public class Common {

    private int value;           // Shared data
    private boolean flag = true; // true: Producer's turn
                                 // false: Consumer's turn


    public synchronized void produce(int i) {
        while (flag == false) {          
            try {
                this.wait();                  // Wait until Consumer consumes
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }

        value = i;
        System.out.println("Producer produced : " + value);

        flag = false;                    // Now it's Consumer's turn
        this.notify();                        // Notify Consumer
    }



    public synchronized int consume() {
        while (flag == true) {           
            try {
                this.wait();                  // Wait until Producer produces
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }


        flag = true;                     // Now it's Producer's turn
        this.notify();                        // Notify Producer

        return value;
    }
}

--Deadlock is a situation where two or more threads are waiting for each other forever, and none of them can continue execution.

Simple Definition:

--Deadlock is a situation where threads are blocked forever waiting for each other’s locks.
